# Product Hub Directory Migration Design

> Date: 2026-05-22
> Scope: `dreambuddy-v2`
> Status: Ready for review

## 1. Goal

本设计用于正式定义中台目录归位与迁移方案。

这次迁移的目标不是重写功能，也不是立即处理所有引用、脚本、构建和导航兼容，而是先把中台总容器、老中台模块归属、文档位置和新模块骨架统一下来，为后续的前端壳构建与模块化落地提供稳定工程基础。

本轮要完成的核心目标只有四个：

1. 创建 `7-产物中台`
2. 把 `7-ARTIFACT_HUB` 整体归入，并重命名为 `系统研究索引体系`
3. 把研究文档剥离到 `docs`
4. 补 `ui-map` 和其他新模块的空目录/壳目录

## 2. Why This Migration Is Needed

当前 `7-ARTIFACT_HUB` 已经不再只是一个临时目录，而是一个具有真实业务价值的独立工程。

但它现在仍然存在两个问题：

- 它的目录语义仍然是旧的 `ARTIFACT_HUB`
- 它实际上承载的是“系统研究索引体系”模块的主要功能实现，却还在被误读成整个中台

如果继续在这个基础上直接新增 `ui-map`、`用户上下文索引系统`、`策略主线` 等模块，会产生三个长期问题：

- 老目录继续承担模糊总容器角色，后续结构越来越混
- 页面模块化和文件模块化无法一一对应
- 新模块只能在旧目录旁边叠加，治理成本会越来越高

因此，应该尽快完成目录归位，让现有工程先在总架构里拥有清晰身份。

## 3. Core Positioning

本轮迁移明确以下定位：

- `7-产物中台` 是新的中台总容器
- 当前 `7-ARTIFACT_HUB` 不是整个中台，而是 `系统研究索引体系` 模块的主要实现
- 新增能力不再挂在一个模糊总目录下，而是在中台总容器内按模块落位

这意味着：

- 老中台不是要被推倒重来
- 老中台是要被归位成新中台架构中的一个已实现模块

## 4. Migration Scope

### 4.1 In Scope

本轮迁移只包括以下四步：

1. 创建 `7-产物中台`
2. 将 `7-ARTIFACT_HUB` 归入该目录，并重命名为 `系统研究索引体系`
3. 将研究文档剥离到 `docs`
4. 创建 `ui-map` 和其他新模块的空目录/壳目录

### 4.2 Out Of Scope

本轮迁移明确不做以下事情：

- 不改现有功能逻辑
- 不立即做后端能力接入
- 不一次性清理所有历史脚本
- 不一次性修完所有构建、引用和导航兼容
- 不在迁移过程中顺手做功能重构

这些内容放到后续模块实现过程中边做边治理。

## 5. The Four Migration Steps

### 5.1 Step 1: Create `7-产物中台`

先创建新的总容器目录：

- `7-产物中台`

它的职责是：

- 承接整个产物中台的总目录语义
- 为后续各模块提供统一归属位置
- 结束 `7-ARTIFACT_HUB` 同时承担“历史工程目录”和“中台总容器”的混乱状态

### 5.2 Step 2: Move `7-ARTIFACT_HUB` Into It And Rename It

将当前：

- `7-ARTIFACT_HUB`

整体移入 `7-产物中台`，并重命名为：

- `系统研究索引体系`

这样做的意义是：

- 现有工程被正式归位为一个模块
- 模块身份从“历史 artifact hub”变成“系统研究索引体系”
- 后续其他模块不需要围绕旧名字去解释自己的归属

### 5.3 Step 3: Move Research Documents Into `docs`

将研究文档从功能实现目录中剥离，统一归档到：

- `7-产物中台/docs`

这样做的原则是：

- 实现目录只放功能实现和支撑脚本
- 文档目录承担研究、方案、规范、FAQ、工程索引等内容

这一步并不要求把所有文档立刻按最细颗粒度重组，但至少要完成“文档与实现分层”。

### 5.4 Step 4: Add Empty Module Directories / Shell Directories

在 `7-产物中台` 下补齐后续要承接的新模块目录。

本轮至少需要建立以下空目录或壳目录：

- `ui-map`
- `用户上下文索引系统`
- `策略主线`
- `系统研究链路`
- `系统运营链路`

这一步的目标不是立刻实现这些模块，而是先完成结构落位，让后续前端壳与功能模块有明确位置，不再边写边找归属。

## 6. Recommended Target Structure

本轮迁移完成后，推荐的总结构如下：

```text
7-产物中台/
  docs/
  ui-map/
  系统研究索引体系/
  用户上下文索引系统/
  策略主线/
  系统研究链路/
  系统运营链路/
```

其中：

- `系统研究索引体系/`
  - 承接原 `7-ARTIFACT_HUB` 的主要实现
- `docs/`
  - 统一放研究文档、FAQ、工程索引、计划与方案文档
- 其他目录
  - 先作为新模块空目录或壳目录存在

## 7. Naming Rules

为了避免后续再次出现目录语义漂移，本轮迁移后应统一遵守以下命名规则。

### 7.1 Top-Level Rule

顶层总容器使用：

- `7-产物中台`

这表示中台总架构，而不是单一模块。

### 7.2 Module Rule

中台内部目录使用模块语义命名，而不是历史项目名或模糊平台名。

例如：

- `系统研究索引体系`
- `用户上下文索引系统`
- `策略主线`
- `系统研究链路`
- `系统运营链路`
- `ui-map`

### 7.3 Documentation Rule

文档统一进入：

- `docs`

如果后续需要更细分，再按模块或文档类型继续细拆，但不再允许研究文档与功能实现长期混放。

## 8. Compatibility Strategy

本轮迁移明确采用：

- `先归位，后兼容治理`

也就是说：

- 先把目录语义和模块边界统一
- 再在后续功能实现过程中逐步处理：
  - 引用修正
  - 脚本修正
  - 构建兼容
  - 导航兼容

这比在迁移阶段一次性把所有问题都处理完更稳，也更符合当前工程实际。

## 9. Why Only Four Steps Are Enough

这四步之所以够用，是因为它们已经完成了最关键的工程骨架搭建：

- 总容器有了
- 老中台归位了
- 文档实现分层了
- 新模块落位了

一旦这四步完成，后续就可以直接进入：

- `ui-map` 前端壳构建
- 模块化功能实现
- 工程索引与 FAQ 建立
- 在功能实现过程中同步处理兼容问题

因此，本轮不需要继续追加更多迁移步骤，否则会把“目录归位”拉成“大规模重构”。

## 10. Risks And Constraints

本轮迁移需要关注以下风险：

- 目录迁移后，原有脚本和相对路径可能暂时失效
- 文档剥离后，历史链接可能需要后续兼容修正
- 新模块目录虽然建立了，但前期仍可能为空目录，需要避免误读为“功能已完成”

这些风险是已知且可接受的，因为它们都可以在后续模块实现过程中逐步消化。

本轮必须坚持的约束是：

- 不把迁移过程变成功能重构
- 不在迁移阶段顺手处理无关逻辑
- 不让老中台继续以“总目录”身份存在

## 11. Final Statement

本轮最终结论明确为：

- 先创建 `7-产物中台`
- 再把 `7-ARTIFACT_HUB` 归入其中，并重命名为 `系统研究索引体系`
- 然后把研究文档剥离到 `docs`
- 最后补 `ui-map` 和其他新模块的空目录/壳目录

四步完成后，即可按文档规划继续做：

- 前端壳构建
- 功能模块逐块实现
- 工程索引和 FAQ 建立

并在后续实现过程中逐步负责：

- 引用兼容
- 脚本兼容
- 构建兼容
- 导航兼容
