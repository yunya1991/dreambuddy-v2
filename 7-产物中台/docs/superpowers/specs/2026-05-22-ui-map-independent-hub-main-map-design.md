# UI-Map Independent Hub Main Map Design

> Date: 2026-05-22
> Scope: `7-ARTIFACT_HUB`
> Status: Ready for review

## 1. Goal

本设计用于正式定义 `ui-map` 作为独立中台首页的第一版主图方案。

这次设计不再把中台与用户前端混在一起讨论，而是明确分开两个系统：

- `dashboard` 是用户前端
- `ui-map` 是中台独立首页

本轮目标不是先改后端，也不是先把所有页面做完，而是先把 `ui-map` 首页的主模块、主关系和横向统一性定清楚。

本轮需要回答的核心问题是：

- `ui-map` 首页第一层到底该按什么模块来展示
- `自定义策略` 与 `系统策略` 如何在首页形成两个清晰来源
- 两类策略如何统一收口进入同一条 `策略主线`
- `用户上下文索引系统` 与 `系统研究索引体系` 如何形成横向关系
- `系统研究链路` 与 `系统运营链路` 如何作为透视层存在

## 2. Problem Statement

之前的方案存在两个方向性问题：

- 容易把中台和 `dashboard` 混成一个系统
- 容易把 `意图闭环`、`AI 推理与推荐层` 直接抬成首页第一层入口，导致首页更像内部机制图，而不是中台主图

这会带来三个问题：

- 用户前端和中台边界不清楚
- 首页物理可视化层级不稳定，看不出平台真正承接的是哪两类策略
- 横向统一性容易被拆散成多个并列模块，而不是一条统一主线和两套协同索引

因此，这次需要把首页主图重新收口为：

- 先看两类策略来源
- 再看统一收口主线
- 再看用户底座与平台能力底座
- 最后看透视体系

## 3. Architecture Decision

本设计选择让 `ui-map` 首页按“来源层 -> 统一主线层 -> 索引底座层 -> 透视层”的方式来组织，而不是做五块平铺模块。

### 3.1 Why First Layer Should Be Strategy Types

`ui-map` 首页第一层明确选用：

- `自定义策略`
- `系统策略`

原因是：

- 这是平台真正承接的两类策略来源
- 这是用户最容易理解的业务入口分法
- 这比把“意图闭环”直接单独抬成首页入口更稳定

首页第一层的目标是先回答：

- 平台承接哪两类策略

而不是先回答：

- 平台内部靠哪些机制生成策略

### 3.2 Custom Strategy Is A Composite Source

`自定义策略` 不是单一按钮，也不是单一对象。

它在中台主图中的真实含义是：

- `意图闭环`
- `AI 推理与推荐`
- 用户个体经验
- 传统联网金融经验

共同形成的策略生成来源。

因此，本设计明确约束：

- `意图闭环 + AI 推理与推荐层` 不再作为首页第一层的独立并列入口
- 它们作为 `自定义策略` 的内部生成机制存在

### 3.3 System Strategy Remains An Independent Source

`系统策略` 继续作为另一类独立来源保留。

它承接的是：

- 系统策略推荐
- 系统研究产物
- 固定研究链路形成的策略结果
- 当前 `feed` 所代表的系统策略入口

但它不是独立后系统。

一旦进入可执行状态，它也必须与 `自定义策略` 一样，统一进入同一条 `策略主线`。

## 4. Unified Mainline

首页第二层明确为：

- `策略主线`

它是这次中台横向统一性的核心收口层。

### 4.1 Why Strategy Mainline Is The Real Unification Layer

统一性不是在来源层完成的，而是在 `策略主线` 这一层完成的。

也就是说：

- `自定义策略` 与 `系统策略` 可以在来源上保持不同
- 但后续不能走两套后系统
- 只要进入执行态，都统一收口到 `策略主线`

### 4.2 Trade Settings As The Convergence Gate

`策略主线` 不是抽象收口，它需要一个明确的收口条件。

本设计明确把收口条件定义为：

- `通过交易设置实现策略收口`

这意味着：

- 上游用户前端 `dashboard` 中的 `交易设置`
- 尤其是其中的 `策略选择`
- 是用户系统与策略系统建立关系的前端绑定点

但这里需要严格区分边界：

- `dashboard` 负责前端入口和关系绑定起点
- `ui-map` 负责中台首页与关系总图
- 中台本体不等于用户前端

### 4.3 Mainline Semantics

`策略主线` 的统一链路继续保持为：

- `策略设置成功 -> 策略任务单 -> 交易链条 -> 交易执行 -> 结果产物 -> 索引`

首页主图不需要把每一层都展开成细节页，但必须明确：

- 两类策略进入这里后不再分叉
- 后续执行与结果都在同一条主线上运行

## 5. Dual Index Foundations

`策略主线` 之后，不能只存在一个笼统索引。

本设计正式定义两套底座索引：

- `用户上下文索引系统`
- `系统研究索引体系`

这两者不是孤立并列关系，而是“用户底座”和“平台基础能力底座”的横向协同关系。

### 5.1 User Context Index System

`用户上下文索引系统` 不是普通归档区，而是用户侧运行底座。

它至少承接以下内容：

- `交易设置`
- `通信设置`
- `策略设置`
- `API 信息设置`
- `用户记忆`
- `专属意识记忆库`

它有两类核心职责。

#### A. 构建期职责

在用户构建策略时，它为 AI 和策略系统提供：

- 用户个体偏好
- 用户历史经验
- 用户专属记忆
- 用户上下文配置

#### B. 执行期职责

在策略进入执行后，它还要在每次执行触发时被调用。

特别是在：

- `1h`
- `4h`
- `1d`

这类周期执行场景中，系统都需要重新读取用户上下文。

这意味着每次执行都不是裸执行，而是：

- `策略任务单 -> 执行触发 -> 用户上下文索引调用 -> 交易执行 -> 结果产物 -> 索引更新`

因此，本设计明确：

- `用户上下文索引系统` 既支撑策略构建
- 也支撑每次执行

### 5.2 System Research Index Foundation

`系统研究索引体系` 保留现有中台从系统研究视角构建出来的能力结构。

它不是普通产物目录，而是平台基础能力层。

它至少承担：

- 系统研究结果沉淀
- 系统策略支撑
- 平台公共能力沉淀
- AI 推理时的优先参考路径来源

也就是说，它的价值不只是“被查看”，而是“被调用”。

### 5.3 Lateral Relationship Between The Two Indexes

这两套索引的关系不是“两边偶尔关联一下”，而是：

- `系统研究索引体系` 提供平台基础能力
- `用户上下文索引系统` 提供用户个体上下文
- AI 和策略系统在生成策略时需要同时调用两者

用户构建策略时的真实组合应明确为：

- `系统研究体系能力`
- `个人经验`
- `AI 推理`
- `传统联网金融经验`

共同形成推荐策略与最优推理路径。

因此，主图中应明确存在以下关系：

- `系统研究索引体系 -> 自定义策略生成`
- `用户上下文索引系统 -> 自定义策略生成`
- `用户上下文索引系统 -> 每次策略执行`

## 6. Visual Main Map Structure

基于以上约束，`ui-map` 首页主图不应做成线性流水图，也不应做成五块平铺卡片，而应做成分层主图。

### 6.1 Layer 1: Strategy Source Layer

第一层为：

- `自定义策略`
- `系统策略`

其中：

- `自定义策略` 内部承载 `意图闭环 + AI 推理与推荐`
- `系统策略` 承接固定研究链路和系统策略入口

### 6.2 Layer 2: Unified Mainline Layer

第二层为：

- `策略主线`

并在视觉上明确标注：

- `通过交易设置实现策略收口`

这句用于解释：

- 前端如何完成用户系统与策略系统关系绑定
- 两类策略如何统一进入同一主线

### 6.3 Layer 3: Index Foundation Layer

第三层不是单线终点，而是两套协同底座：

- `用户上下文索引系统`
- `系统研究索引体系`

这层的关键视觉约束是：

- `用户上下文索引系统` 不能只画成主线后面的结果框
- 它必须同时连向策略构建与策略执行
- `系统研究索引体系` 主要连向自定义策略生成与平台推荐能力

### 6.4 Layer 4: Perspective Layer

第四层为透视层：

- `系统研究链路`
- `系统运营链路`

它们的职责不是生产对象，而是展示关系。

#### System Research Chain

用于展示：

- 固定研究流程
- 系统策略形成过程
- 系统研究产物关系

#### System Operations Chain

用于展示：

- 用户前端如何进入
- 策略如何完成收口
- 用户上下文何时被调用
- 策略执行如何发生
- 结果如何进入索引体系

## 7. Homepage Expression Rules

为了避免首页再次滑向“内部机制图”或“平铺模块页”，本设计增加以下可视化约束。

### 7.1 Do Not Flatten All Modules

首页不得把以下模块做成同级平铺：

- `意图闭环`
- `AI 推理与推荐`
- `自定义策略`
- `系统策略`
- `策略主线`

这样会破坏层级关系。

### 7.2 Expose Sources, Hide Internal Mechanics One Level Deeper

首页第一眼只暴露：

- 两类策略来源
- 一条统一主线
- 两套索引底座
- 两类透视链路

至于：

- `意图闭环`
- `AI 推理与推荐`
- 具体对象层

这些内容应下沉到下一层模块详情中再展开。

### 7.3 User Context Index Must Be Shown As Runtime Foundation

首页必须明确表达：

- `用户上下文索引系统` 既参与策略构建
- 又参与每次执行

它不是结果归档框，而是运行底座。

## 8. Success Criteria

这份首页主图方案成立后，应满足以下判断标准：

- `ui-map` 被明确为独立中台首页，而不是用户前端页面
- 首页第一层先展示 `自定义策略 / 系统策略`
- `意图闭环 + AI 推理与推荐` 被收进 `自定义策略` 内部
- 两类策略统一收口到同一条 `策略主线`
- `交易设置` 被明确为策略收口与用户绑定条件，但不把中台与前端混为一体
- `用户上下文索引系统` 被定义为用户底座，而不是后置归档区
- `系统研究索引体系` 被定义为平台基础能力底座
- 两套索引之间的横向关系被明确表达
- `用户上下文索引系统` 同时连向策略构建与策略执行
- `系统研究链路` 与 `系统运营链路` 被定义为透视层，而不是主线本体

## 9. Final Statement

本轮最终结论明确为：

- `ui-map` 是独立中台首页
- 首页第一层按 `自定义策略 / 系统策略` 两类来源组织
- `自定义策略` 内部承载 `意图闭环 + AI 推理与推荐`
- 两类策略通过交易设置完成策略收口，统一进入 `策略主线`
- `用户上下文索引系统` 与 `系统研究索引体系` 共同作为中台底座
- 其中 `用户上下文索引系统` 既参与策略构建，也参与每次执行
- 最终由 `系统研究链路` 与 `系统运营链路` 统一展示上述关系
