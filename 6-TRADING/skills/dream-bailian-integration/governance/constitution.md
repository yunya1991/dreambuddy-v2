## 最高原则

### §1 第一性原理
> **"市场总是沿着阻力最小方向运行，趋势具有延续性"**

所有百炼集成决策必须服务于这一核心原理。

### §2 双脑协同
- **左脑**: 确定性规则、门禁、风险阈值 → 执行
- **右脑**: 情景推演、概率判断、盘感 → 建议
- **仲裁**: 左脑 fail → 直接拒绝；左脑 warn + 右脑高置信 → shadow_only

### §3 证据驱动
- 每条关键规则必须有 evidence_refs 可追溯
- 不允许根据猜测补全缺失字段
- 证据不能追溯到 material_refs → reject

## 百炼集成宪法

### §4 API调用原则
1. **免费额度优先**: 优先使用免费模型 (qwen-plus)
2. **按需升级**: 免费额度不足时切换到付费模型
3. **成本控制**: 每次调用必须记录 token 消耗
4. **幂等设计**: 重复调用不产生副作用

### §5 知识库RAG原则
1. **数据隔离**: 交易数据、治理数据、文档数据分类存储
2. **权限控制**: 敏感数据需要 AAM 验证
3. **版本管理**: 知识库变更必须记录版本号
4. **质量门禁**: RAG 检索结果必须通过 distillation-quality-gate

### §6 工作流编排原则
1. **无状态设计**: 工作流步骤必须可重试
2. **错误恢复**: 每步必须有回滚方案
3. **可观测性**: 关键步骤必须有日志记录
4. **超时控制**: 每步必须有超时设置

### §7 Function Calling原则
1. **安全优先**: 危险操作需要二次确认
2. **参数校验**: 输入参数必须符合 schema
3. **错误处理**: 调用失败必须有降级方案
4. **审计追踪**: 所有调用必须记录 trace_id

### §8 产物传递原则 (AAM)
1. **双通道投递**: 秘书邮箱 + 前端产物中心
2. **frontmatter完整**: title, department, chain_phase, date, type, status, tags, by_a_phase
3. **验证通过**: curl 验证返回 200
4. **日志记录**: 投递操作必须记录到日志

### §9 合规审查原则
1. **P0/P1变更**: 必须人工确认
2. **影子验证**: 重大变更先影子执行
3. **回滚方案**: 每项变更必须有 rollback_plan_id
4. **变更审计**: 所有变更必须记录到审计日志

## 决策矩阵

| 场景 | 左脑判定 | 右脑置信 | 决策 |
|:---|:---:|:---:|:---|
| 正常执行 | pass | 高 | ✅ execute |
| 警告执行 | warn | 高 | ⚠️ warn_execute |
| 禁止执行 | fail | - | ❌ reject |
| 高风险 | warn | 低 | 🔒 shadow_only |
| 低置信 | pass | 低 | ⏸️ observe |

---

## 禁止事项 (Hard Bans)

1. ❌ 禁止在未配置 API Key 的情况下调用百炼
2. ❌ 禁止暴露 API Key 到日志或产物中
3. ❌ 禁止在生产环境使用 shadow_only 策略
4. ❌ 禁止删除带有 "禁忌:不可删" 标记的记忆
5. ❌ 禁止在无 rollback_plan_id 的情况下发布变更
6. ❌ 禁止绕过合规审查直接执行 P0/P1 变更
