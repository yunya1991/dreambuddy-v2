# 7-ARTIFACT-HUB-V2 中枢治理规范

> 版本：v1.1  
> 更新日期：2026-05-16  
> 作用：定义 `7-ARTIFACT-HUB-V2` 作为公司中枢时的治理边界、决策分级标记、审批门、审计要求与 fail-closed 规则。

## 1. 目标

本规范用于回答一个核心问题：

> 当 `7-ARTIFACT-HUB-V2` 从“产物中台 + 路由服务”升级为“公司治理架构驱动的 AI 中枢”后，系统如何在组织层、决策层、执行层、分发层和审计层保持可控？

本规范关注：

- 决策分级标记；
- 董事会（治理委员会）机制；
- 人工审批门；
- fail-closed 原则；
- 审计与证据链；
- 外部分发与高风险动作治理。

## 2. 治理原则

### 2.1 解释优先

对关键动作，系统必须先能解释，再允许执行。

关键动作包括：

- 高风险交易；
- 面向外部分发；
- 关键路由调整；
- 跨部门职责变更；
- 重大组织优化。

### 2.2 审计优先

关键动作必须留下证据链。

最小审计要求：

- `trace_id`
- `decision`
- `execution`
- `artifact`
- `distribution`
- `review`

### 2.3 组织优先

系统不是纯技术系统，所有关键动作都要映射到部门职责：

- 谁发起；
- 谁负责；
- 谁审批；
- 谁复盘。

### 2.4 Fail-Closed 优先

对高风险动作，只要关键条件不满足，就默认拒绝执行。

不允许“先放过去再说”。

## 3. 治理分层

系统治理分为 4 层：

### 3.1 Constitution Layer

最顶层约束，定义不可突破的原则。

包括：

- 风险红线；
- 审批红线；
- 对外分发红线；
- 关键权限红线；
- 审计缺失红线。

### 3.2 Governance Layer

定义治理策略、路由策略、审批策略与可执行范围。

包括：

- `policy_version`
- 决策分级标记规则
- 部门职责边界
- 升级规则
- 回滚规则

### 3.3 Board Layer

六人董事会（治理委员会）层。

负责：

- 监督；
- 协调；
- 处理中问题；
- 形成重大事项议案。

### 3.4 Execution Layer

路由、任务、结果、产物、分发等执行层。

其职责不是自己决定一切，而是在治理层之下执行。

## 4. 决策分级标记

当前阶段，`L1 / L2 / L3` 的定位是治理标记字段，而不是已经落地的自动执行流程。

也就是说，本规范当前只要求系统能够：

- 为决策或议案打上分级标记；
- 根据分级决定是否需要会签或人工审批；
- 在审计与 UI 中显示分级结果。

本规范当前不假设以下能力已经实现：

- 自动升级工作流；
- 自动会签编排；
- 自动触发审批 UI；
- 独立的部长 Agent 执行引擎。

### 4.1 L1 小问题

`L1` 表示可以由部门内处理的低风险事项。

典型场景：

- 内容标签修正；
- 非核心路由调整；
- 普通投放节奏调整；
- 普通摘要归档；
- 普通异常重试。

当前要求是给记录打上 `decision_level=L1` 标记，而不是要求系统已实现自动自主处置流程。

### 4.2 L2 中问题

`L2` 表示需要跨部门协调或会签的中等级事项。

典型场景：

- 跨部门流程重排；
- `/chain` 两套工作流优先级调整；
- 产物可见性范围变化；
- 分发策略切换；
- 组织绩效预警后的局部优化。

当前要求是给记录打上 `decision_level=L2` 标记，并在治理侧保留建议方案与证据链。

### 4.3 L3 重大事项

`L3` 表示必须进入人工审批门的重大事项。

典型场景：

- 核心交易策略变更；
- 高风险资金动作；
- 外部重大投放；
- 部门职责重构；
- 关键权限调整；
- 董事会建议的人事优化与组织重组。

当前要求是给记录打上 `decision_level=L3` 标记，并明确其不得绕过人工审批。

## 5. 六人董事会（治理委员会）

### 5.1 成员

- 研究部长 Agent
- 交易部长 Agent
- 治理部长 Agent
- 运营部长 Agent
- HR 部长 Agent
- 市场部长 Agent

### 5.2 职责

- 监督部门目标；
- 发现低效与异常；
- 推动跨部门协同；
- 参与 `L1` / `L2` / `L3` 标记判断；
- 讨论中问题；
- 形成重大事项议案。

### 5.3 不做什么

董事会不是绕过治理层直接下命令。

它不能：

- 无审批执行重大事项；
- 绕过审计链；
- 绕过部门职责边界；
- 绕过 fail-closed 规则。

## 6. 人工审批门

### 6.1 审批触发条件

以下情况必须进入人工审批门：

- `decision_level = L3`
- 高风险交易
- 大规模外部分发
- 核心策略切换
- 部门重组或人事优化
- 无法给出完整 evidence 的关键动作

### 6.2 审批票据最小字段

审批票据建议至少包含：

- `proposal_id`
- `source_department`
- `initiator_agent_id`
- `decision_level`
- `problem_summary`
- `evidence_refs`
- `recommended_action`
- `expected_impact`
- `rollback_plan`
- `approval_status`

### 6.3 审批结果

审批结果至少包括：

- `approved`
- `rejected`
- `needs_more_evidence`

### 6.4 审批拒绝后的规则

若审批拒绝：

- 不得执行最终动作；
- 不得发布 promotion 结果；
- 必须保留拒绝原因；
- 如已执行过可逆动作，必须进入回滚流程。

## 7. 审计与证据链

### 7.1 最小证据链

对任何关键动作，至少要能追到：

- 谁发起；
- 为什么这样决策；
- 执行了什么；
- 产出了什么；
- 发给了谁；
- 结果如何；
- 谁审核过；
- 谁批准过。

### 7.2 最小追踪字段

建议所有核心对象逐步统一：

- `trace_id`
- `workflow_id`
- `workflow_type`
- `department`
- `policy_version`
- `schema_version`
- `producer`
- `timestamp`

### 7.3 回放要求

对关键链路，系统必须能：

- 按 `trace_id` 回放；
- 按 `workflow_id` 聚合；
- 按 `department` 检索；
- 按 `proposal_id` 看审批与执行关系。

## 8. Fail-Closed 规则

### 8.1 必须 Fail-Closed 的场景

以下场景默认 fail-closed：

- 高风险交易缺证据；
- `workflow_id` 缺失但进入关键链路；
- `trace_id` 缺失但进入关键决策；
- 面向外部分发缺审批；
- 重大事项缺 proposal；
- 缺少 rollback plan 的重大动作；
- 决策与执行对象不一致；
- 审计对象缺失。

### 8.2 可降级但不拒绝的场景

以下场景可先标记 warning：

- 摘要信息不完整；
- 低风险内容缺少部分标签；
- 内部只读页面信息延迟；
- 非关键统计数据缺失。

### 8.3 最小实现路径

当前阶段建议先集中实现一个 `failClosedCheck()` 之类的共享校验函数，而不是把判断逻辑散落在多个页面或路由中。

该函数至少检查：

- `trace_id` 是否存在；
- `workflow_id` 是否存在；
- `policy_version` 是否存在；
- `decision_level=L3` 时是否具备审批状态；
- 关键动作是否具备 `evidence_refs`；
- 重大动作是否具备 `rollback_plan`。

建议接入位置：

- Hub 路由入口；
- 高风险分发入口；
- 交易关键执行入口；
- 后续审批票据写入入口。

### 8.4 错误语义建议

建议后续统一 reason codes，例如：

- `missing_trace_id`
- `missing_workflow_id`
- `missing_policy_version`
- `approval_required`
- `evidence_incomplete`
- `rollback_plan_required`
- `distribution_scope_denied`

## 9. 分发治理

### 9.1 外部分发必须建模

市场化中台第一阶段虽然主要服务内部销售/运营，但它会承担对外运营动作。

因此分发动作必须被视为一等治理对象。

### 9.2 分发最小字段

建议对每条分发记录至少保留：

- `distribution_id`
- `artifact_id`
- `target_segment`
- `channel`
- `delivery_policy`
- `reason`
- `status`
- `result`

### 9.3 高风险分发

以下情况必须进入强化治理：

- 面向外部大量触达；
- 涉及交易策略或投资建议；
- 涉及品牌级对外表达；
- 涉及跨渠道同步。

## 10. 绩效与进化治理

### 10.1 HR 视角

在目标治理模型中，HR 将负责：

- 绩效评估；
- 资源使用评估；
- 招聘建议；
- 组织优化建议。

因此，后续应让所有关键链路能够回流到：

- 部门绩效；
- 流程效率；
- 资源消耗；
- 收益/成本；
- 优化建议。

当前阶段仅建议保留最小指标类别，不在本规范中展开完整 KPI 模型。第一版可先聚焦：

- 交易部：盈亏、命中率、异常率；
- 研究部：产物产出量、复用率；
- 治理部：异常发现率、回放覆盖率；
- 运营部：触达量、转化率、投放异常率。

### 10.2 市场部视角

在目标治理模型中，市场部将负责：

- 外部赛道观察；
- GitHub 热门项目；
- 热门策略；
- KOL；
- 热门币种；
- 外部机会和威胁输入。

后续应让这些情报能够进入：

- 研究部；
- 交易部；
- 运营部；
- 董事会讨论。

## 11. 文档与实现边界

本规范定义的是“治理规则与边界”，不是具体 UI 或表结构实现。

它约束：

- README 的系统定位；
- Query / Route / Trace / Event / Task / Result 契约；
- `/feed`、`/chain`、`/ops`、市场化中台的治理边界；
- 董事会与审批机制；
- 对象模型落地方向。

它当前不等于：

- 已实现的审批 UI；
- 已实现的董事会控制台；
- 已实现的自动升级流程；
- 已实现的市场化中台执行引擎。

## 12. 一句话结论

`7-ARTIFACT-HUB-V2` 的治理规范不是为了让系统更复杂，而是为了确保：

> 当这套系统逐步演进为“公司中枢”时，AI 的判断、执行、分发与优化始终处在可解释、可控制、可审计、可审批的边界之内。
