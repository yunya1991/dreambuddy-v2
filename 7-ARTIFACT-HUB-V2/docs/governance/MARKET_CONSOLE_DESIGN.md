# 市场化中台设计

> 版本：v1.1  
> 更新日期：2026-05-16  
> 状态：规划中  
> 作用：定义市场化中台的定位、页面职责、治理边界与与研究中台/治理控制台的关系。

## 0. 当前实现状态

市场化中台当前仍是规划方案，主仓尚无对应页面实现。

| 项目 | 状态 | 说明 |
|---|---|---|
| 市场化中台页面入口 | 规划中 | 当前主仓尚无对应路由或独立应用 |
| 技术选型 | 规划中 | 默认推荐同仓双入口方案 |
| 用户分层 / 推送分发 / 投放管理 | 规划中 | 本文档描述目标能力，不代表当前可运行 |

## 1. 定位

市场化中台不是对外客户平台，而是第一阶段面向内部销售/运营团队的中台入口。

它服务于：

- 内部销售；
- 运营团队；
- 渠道运营；
- 内容分发与投放管理。

它在公司中枢中的定位是：

> 运营部主视角的分发治理平台。

## 2. 技术归属

当前默认方案是“同仓双入口”，即：

- 与研究中台共享一套底层能力；
- 优先落在同一前端仓位中扩展新入口；
- 第一阶段更偏向在现有前端主线中新增路由分组，而不是单独再起一套完全独立系统。

本文件当前不主张把它写成已经存在的独立应用。

## 3. 核心目标

第一阶段市场化中台的核心目标是：

- 做内容路由；
- 做用户分层；
- 做推送分发；
- 做投放管理；
- 做效果反馈；
- 做分发审计。

它不是“营销官网”，而是“运营分发操作台”。

## 4. 要解决的问题

### 4.1 分发黑箱

系统必须能回答：

- 哪些内容被分发给了哪些人；
- 为什么是这些人；
- 为什么走这个渠道；
- 为什么是这个时间点；
- 分发后发生了什么。

### 4.2 运营执行碎片化

当前典型问题包括：

- 内容池与策略池割裂；
- 用户分层标准不统一；
- 推送渠道和投放记录分散；
- 无法把 route/trace 与运营动作关联起来。

### 4.3 运营动作缺少审计

对于以下动作，必须能回溯：

- 推送谁发起的；
- 使用了哪条内容；
- 命中了哪类人群；
- 用了什么策略；
- 最终反馈如何。

## 5. 市场化中台页面结构

### 5.1 首页

首页应聚焦运营分发，不做内容门户化。

建议分为三列：

- 左侧：用户分层 / 筛选 / 渠道切换
- 中间：内容池 / 策略池 / 分发任务
- 右侧：投放状态 / 推送结果 / 风险与告警

### 5.2 核心页面

建议至少包含：

- `内容池`
- `用户分层`
- `内容路由`
- `推送分发`
- `投放管理`
- `效果反馈`
- `分发审计`

## 6. 关键模块

### 6.1 内容池

规划中的内容池模块将用于统一承接可用于分发的内容。

来源包括：

- 研究中台产物；
- 策略库；
- 市场情报摘要；
- 运营包装后的内容版本。

### 6.2 用户分层

用户分层是市场化中台的核心能力之一。

第一阶段不要求建立复杂 CRM，但至少需要支持：

- 按运营状态分层；
- 按渠道来源分层；
- 按兴趣主题分层；
- 按风险级别分层。

### 6.3 内容路由

规划中的内容路由模块将用于回答：

- 什么内容可以发；
- 发给谁；
- 通过什么渠道；
- 谁批准；
- 是否命中红线。

### 6.4 推送分发

分发动作可能包括：

- TG 推送；
- Twitter/X 运营动作；
- 内部销售触达任务；
- 渠道内容投放。

分发本身必须被建模为治理对象，而不是只当作“发送成功”。

### 6.5 投放管理

投放管理关注：

- 分发任务状态；
- 渠道消耗；
- 资源使用；
- 转化结果；
- 异常中断。

### 6.6 效果反馈

目标设计中，需要把反馈结果重新回流到：

- 运营部；
- HR 绩效；
- 市场部情报；
- 董事会；
- 治理部审计。

### 6.7 分发审计

目标设计中，对于所有关键分发动作，应能按：

- `distribution_id`
- `artifact_id`
- `trace_id`
- `channel`
- `target_segment`

进行回放与追溯。

## 7. 与其他中台的关系

### 7.1 与研究中台

- 在目标协作关系中，研究中台将提供高价值内容；
- 市场化中台将把这些内容转化为可路由、可投放、可审计的运营对象。

### 7.2 与治理控制台

- 在目标协作关系中，治理控制台将承接 route / trace / approval / audit；
- 市场化中台将利用这些治理能力完成运营分发。

### 7.3 与 `/chain`

- 在目标协作关系中，`/chain` 将提供交易链路与交易结果的可见性；
- 市场化中台将只消费经治理允许外发的内容与结果，不直接替代交易监控。

### 7.4 与 HR

- 市场化中台的资源消耗、投放效率、触达质量后续将进入绩效评估；
- HR 可基于这些结果做组织优化与招聘建议。

### 7.5 与市场部

- 在目标协作关系中，市场部将提供外部竞争情报；
- 市场化中台将把这些情报转化为实际运营分发动作。

## 8. 治理边界

市场化中台不允许绕过以下治理边界：

- 未审批的高风险外部分发；
- 无 trace 的关键分发动作；
- 无 evidence 的策略外发；
- 无 target_segment 规则的群发行为；
- 不可回溯的运营投放。

## 9. 对象模型映射

市场化中台主要依赖这些对象：

- `Artifact`
- `Distribution`
- `Decision`
- `Audit`
- `Performance`
- `MarketIntel`
- `ApprovalGate`

其中 `Performance`、`MarketIntel`、`ApprovalGate` 当前仍属于后续阶段对象，不代表已完成实现。

## 10. 演进方向

### Phase 1

先做内部销售/运营的分发治理平台：

- 内容池；
- 用户分层；
- 分发任务；
- 投放管理；
- 分发审计。

### Phase 2

增强对外运营：

- TG / Twitter / 渠道动作联动；
- 更丰富的用户分层；
- 更多渠道反馈。

### Phase 3

逐步向外部客户平台演化，但前提是：

- 审计与审批机制成熟；
- 风险边界稳定；
- 黑箱治理能力足够强。

## 11. 一句话结论

市场化中台的本质不是“运营页面”，而是：

> 让运营部能够在公司中枢治理边界内，把研究内容、市场情报和策略资产转化成可路由、可投放、可回放的分发动作。
