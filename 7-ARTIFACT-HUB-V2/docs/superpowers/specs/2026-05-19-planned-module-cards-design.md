# Planned 页模块卡片增强设计

## 1. 背景

当前 `/planned` 占位页已经具备：

- `Coming Soon` 状态提示
- `所属模块`
- `预计承接能力`
- 返回 `/ui-map?node=...` 并自动打开对应 Drawer

但页面内容仍偏统一模板，模块之间的差异化还不够强。用户希望继续增强不同模块的 `planned` 页，让页面一眼能看出：

- `Feed` 更偏内容门户结构
- `Query Layer` 更偏查询对象与数据能力
- `Data Fabric` 更偏底层目录、元数据与任务结果协议

## 2. 目标

在不把 `/planned` 做成重型文档页的前提下，将其升级为“平衡型模块卡片”页面：

- 保持当前占位页的轻量感和可读性
- 让不同模块具备清晰的内容重点
- 维持统一样式框架，避免三个模块完全割裂成不同页面

## 3. 设计选择

最终采用可视化对比中的 **B 方案：平衡型模块卡片**。

原因：

- 比轻量列表更有层次，模块差异更明确
- 比重型说明板更克制，仍适合 `planned` 占位页定位
- 信息量适中，既能说明未来方向，也不会抢走真正实现页的角色

## 4. 页面结构

`/planned` 页面在保留现有头部区块的基础上，新增一个模块说明卡片区。

页面层次：

1. 顶部状态区
   - `Planned Route`
   - `coming soon`
   - 当前入口
   - 所属模块
   - 目标路径

2. 模块卡片区
   - 根据模块渲染一张主卡片
   - 卡片采用左右两列结构

3. 返回与导航区
   - `Back to Previous Node`
   - `Back to UI Map`
   - `Open Ops Home`

## 5. 模块卡片设计

### 5.1 Feed

卡片结构：

- 左列：内容门户结构
- 右列：预计承接能力

左列内容示例：

- `/feed` 首页
- `/feed/[category]`
- `/feed/[category]/[artifactId]`

右列内容示例：

- 内容浏览
- 分类聚合
- 上下文阅读

目标：让用户明确它未来是“页面入口层 + 内容阅读层”。

### 5.2 Query Layer

卡片结构：

- 左列：查询对象
- 右列：数据能力

左列内容示例：

- `ArtifactIndex`
- `ArtifactContent`
- `SearchResult`

右列内容示例：

- 分类映射
- 摘要提取
- 标签与 phase 聚合

目标：让用户明确它未来是“内容查询与聚合能力层”，而不是直接面向最终用户的页面入口。

### 5.3 Data Fabric

卡片结构：

- 左列：底层目录
- 右列：协议与运行数据

左列内容示例：

- `artifacts`
- `meta`
- `config`

右列内容示例：

- `task_*.json`
- `result_*.json`
- archive / registry

目标：让用户明确它未来是“底层真相源与运行协议层”。

## 6. 交互与上下文

保留并延续当前行为：

- 规划中链接继续带 `target / label / node / module`
- `Back to Previous Node` 返回 `/ui-map?node=...`
- `/ui-map` 在页面加载时自动读取 `node` 参数并打开对应 Drawer

本次增强只改变 `/planned` 的内容呈现，不改变跳转协议。

## 7. 实现边界

本次只增强 `planned` 页展示，不做以下内容：

- 不实现真实 `/feed` 页面
- 不新增真实 Query Layer 页面
- 不扩展真实 Data Fabric 浏览器
- 不引入前端框架或额外依赖

继续保持：

- Express 路由
- 服务端 HTML 字符串渲染
- 最小量 CSS

## 8. 验收标准

- `Feed` 模块页明确出现“内容门户结构”与“预计承接能力”双列信息
- `Query Layer` 模块页明确出现“查询对象”与“数据能力”双列信息
- `Data Fabric` 模块页明确出现“底层目录”与“协议/运行数据”双列信息
- 页面继续包含所属模块、目标路径、返回上一个节点入口
- `/ui-map?node=...` 的节点恢复行为不回退
- 现有测试继续通过，并新增覆盖模块卡片内容差异化的断言
