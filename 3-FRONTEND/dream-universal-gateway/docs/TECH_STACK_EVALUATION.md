# Dream Universal Gateway — 前端技术栈深度评估报告

> 日期: 2026-05-13 | 目标: 面向批量用户，一次选型到位，避免返工
> 评估范围: 8大维度 × 多规模场景验证

---

## 0. 核心结论

| 维度 | 推荐方案 | 理由 |
|------|----------|------|
| **框架** | **Next.js 15.x** (App Router) | 15.x已稳定(15.4+), PPR+Turbopack性能更优, 长期支持 |
| **部署** | **自托管Docker + 腾讯云CVM** | SSE无超时限制, 完全可控, 成本可预期 |
| **认证** | **Auth.js v5 + Prisma + PostgreSQL** | 自托管零成本, 支持中国手机号/微信OAuth, 无厂商锁定 |
| **支付** | **聚合支付SDK(PayJS/柠檬) → 企业商户资质** | 微信/支付宝双通道, 个人开发者可先PayJS, 规模化后切换直连 |
| **实时通信** | **SSE + Redis Pub/Sub** | 对话场景单向流即可, 自托管无超时, Redis解决多实例同步 |
| **状态管理** | **Zustand 5.x** | 轻量, SSR友好, 批量用户场景性能优 |
| **国际化** | **next-intl** | App Router原生支持, 按需加载语言包 |
| **监控** | **Sentry + 自建日志** | 错误追踪+性能APM, 批量用户问题可定位 |

## 1. 框架选型: Next.js 14 vs 15

### 1.1 对比

| 维度 | Next.js 14 | Next.js 15.x |
|------|-----------|-------------|
| App Router | 稳定 | **更成熟**, 缓存策略优化 |
| Turbopack | 实验性 | **生产可用**, 构建速度提升70% |
| PPR (Partial Prerendering) | 无 | **实验性但可用**, 静态shell+动态流 |
| Server Actions | 稳定 | **增强**, withRedirect等新API |
| Node.js要求 | ≥18.17 | **≥18.18**, 推荐20.x |
| 社区生态 | 成熟 | **快速追赶**, 主流库已适配 |
| 长期支持 | 到15.x发布 | **当前主线**, 至少2年+ |

### 1.2 决策: Next.js 15.x

**理由**:
1. 15.4已进入稳定期, 非早期preview
2. Turbopack生产可用, 开发体验和构建性能显著提升
3. PPR对批量用户的首屏加载体验很重要
4. 新项目没有14→15迁移成本, 直接用最新稳定版
5. 产物中台用14, Gateway用15, 互不影响(独立项目)

### 1.3 ⚠️ 风险与对策

| 风险 | 概率 | 对策 |
|------|------|------|
| 15.x minor breaking change | 低 | 锁定`15.x.y`精确版本, 非tilde |
| 某些UI库尚未适配15 | 中 | 优先使用Headless UI (Radix/shadcn) |
| PPR正式版API变更 | 中 | 初期不用PPR, Phase 4再启用 |

## 2. 部署架构: 自托管 vs Vercel

### 2.1 对比

| 维度 | Vercel | 自托管Docker + 腾讯云 |
|------|--------|---------------------|
| **SSE超时** | Edge 25s / Serverless 60s **硬限制** | **无限制** |
| **WebSocket** | ❌ 不支持 | ✅ 完全支持 |
| **多实例同步** | 需Redis Pub/Sub (额外成本) | 需Redis但**同机房零延迟** |
| **成本可预期性** | 按请求计费, 批量用户费用不可控 | **固定月费**, 可预期 |
| **数据合规** | 服务器在海外 | **国内机房**, 延迟低, 合规 |
| **定制性** | 有限(vendor lock-in) | **完全可控** |
| **运维成本** | 零 | 需要基础运维能力 |

### 2.2 决策: 自托管Docker + 腾讯云CVM

**理由**:
1. **SSE是核心交互**, Vercel的25-60s超时对长分析链路是硬伤
2. 批量用户场景, Vercel按请求计费**成本失控风险高**
3. 腾讯云CVM(你已有49.233.123.96), **同机房部署产物中台+Gateway+Redis**
4. 国内用户访问腾讯云**延迟更低**(<50ms vs 海外200ms+)
5. 金融应用数据不出境, **合规性更好**

### 2.3 部署架构图

```
腾讯云CVM (49.233.123.96)
├── Nginx (反向代理 + SSL + HTTP/2)
│   ├── :3456 → 产物中台 (内部)
│   ├── :3457 → Gateway (用户端)
│   └── :6379 → Redis (内网)
├── PM2 / Docker Compose
│   ├── product-hub (现有)
│   └── gateway (新增)
└── PostgreSQL (用户数据/积分/会话)
```

### 2.4 扩容路径

| 用户量 | 架构 | 成本/月 |
|--------|------|---------|
| 0-500 | 单CVM + Docker | ~¥200 |
| 500-5K | CVM升配(4C8G) + Redis | ~¥500 |
| 5K-50K | CVM集群(2-3台) + SLB + Redis集群 | ~¥2000 |
| 50K+ | K8s + RDS + Redis集群 + CDN | ~¥5000+ |

## 3. 用户认证: Auth.js vs Clerk vs Supabase Auth

### 3.1 对比

| 维度 | Auth.js v5 | Clerk | Supabase Auth |
|------|-----------|-------|--------------|
| **自托管** | ✅ 完全开源 | ❌ 仅云端 | ✅ Docker |
| **成本(10K MAU)** | ¥0 | ¥0 | ¥0 |
| **成本(100K MAU)** | ¥0 | ~¥1,080/年 | ~¥300/年 |
| **中国手机号认证** | ✅ 自建SMS适配器 | ⚠️ 需确认 | ⚠️ 需确认 |
| **微信OAuth** | ✅ 自建Provider | ❌ 不支持 | ❌ 不支持 |
| **多租户** | ❌ 需自建 | ✅ 内置 | ❌ 需自建 |
| **Next.js App Router** | ✅ 完整支持 | ✅ 一行中间件 | ✅ 支持 |
| **厂商锁定** | ❌ 无 | ⚠️ 高 | ⚠️ 中 |
| **迁移成本** | 低 | **极高**(全员重新认证) | 中 |

### 3.2 决策: Auth.js v5 + Prisma + PostgreSQL

**理由**:
1. **必须支持微信登录和手机号验证码** — Clerk/Supabase对中国认证支持弱
2. **自托管零成本**, 批量用户不产生认证费用
3. **无厂商锁定**, 认证是"最昂贵的架构决策之一", 一旦选择很难切换
4. Prisma + PostgreSQL可灵活扩展用户模型(积分/套餐/策略配置等)
5. 你已有PostgreSQL经验, 腾讯云CVM可直接部署

### 3.3 认证方案设计

```
认证方式:
├── 手机号+验证码 (主推, 覆盖所有中国用户)
│   └── SMS Provider: 腾讯云SMS / 阿里云SMS
├── 微信扫码登录 (P1, 提升转化率)
│   └── 微信开放平台OAuth2.0
└── 邮箱+密码 (P2, 国际用户备选)
```

### 3.4 ⚠️ 返工风险

| 风险 | 对策 |
|------|------|
| 手机号验证码成本 | 腾讯云SMS ¥0.045/条, 10K用户注册约¥450 |
| 微信OAuth需要企业资质 | 初期可只用手机号, 微信登录Phase 4加 |
| 验证码安全(轰炸) | 图形验证码前置 + 频率限制(1分钟1次) |

## 4. 支付集成: 积分充值

### 4.1 微信/支付宝接入方案

| 阶段 | 方案 | 要求 | 成本 |
|------|------|------|------|
| **MVP(0-1K用户)** | PayJS聚合支付 | 个人开发者可用 | 费率1.0% |
| **成长期(1K-10K)** | 企业商户直连 | 营业执照+对公账户 | 费率0.6% |
| **成熟期(10K+)** | 自建支付中台 | 金融资质 | 费率0.38% |

### 4.2 支付流程

```
用户选择套餐 → [后端创建订单] → 调用支付SDK获取支付参数
  → 前端调起微信/支付宝收银台 → 用户完成支付
  → 支付平台回调后端 → 后端验签+更新积分
  → SSE推送充值结果到前端
```

### 4.3 ⚠️ 返工风险

| 风险 | 对策 |
|------|------|
| PayJS个人资质限制 | 先用PayJS跑通, 同时准备企业资质 |
| 支付回调安全 | 验签+幂等处理+金额校验 |
| 订单与积分一致性 | 数据库事务, 失败自动回滚 |
| Apple Pay需要额外接入 | Phase 2再加, MVP只做微信/支付宝 |

## 5. 实时通信: SSE vs WebSocket

### 5.1 决策: SSE + Redis Pub/Sub

**理由**:
1. 对话场景**单向流**即可(服务端→客户端), SSE天然适合
2. 自托管**无超时限制** (Vercel的25-60s限制不存在)
3. SSE比WebSocket实现简单, 调试方便
4. Redis Pub/Sub解决多实例消息同步问题

### 5.2 架构

```
用户请求 → Nginx → Next.js API Route (SSE)
                         │
                    Redis Pub/Sub
                    (跨实例广播)
                         │
              ┌──────────┼──────────┐
              │          │          │
         Instance 1  Instance 2  Instance 3
         (User A)    (User B)    (User C)
```

### 5.3 批量用户性能指标

| 用户量 | SSE连接数 | Redis内存 | 推荐Redis配置 |
|--------|----------|-----------|-------------|
| 100 | 100 | ~5MB | 单实例 |
| 1K | 1K | ~50MB | 单实例 |
| 10K | 10K | ~500MB | 主从 |
| 50K | 50K | ~2.5GB | 集群(3主3从) |

### 5.4 ⚠️ 关键优化

| 优化项 | 说明 |
|--------|------|
| 心跳保活 | 每15s发送`: heartbeat\n\n`, 防止连接断开 |
| 自动重连 | 客户端`EventSource.onerror` → 3s后重连 + 恢复上下文 |
| HTTP/2 | Nginx启用HTTP/2, 解决6连接/域名限制 |
| 虚拟滚动 | 消息列表使用`react-window`, >100条消息时只渲染可视区域 |
| 消息去重 | 使用`Set<messageId>`防止重连后重复消息 |

## 6. 状态管理: Zustand 5.x

### 6.1 决策: Zustand 5.x

**理由**:
1. 极轻量(~1KB), 批量用户场景bundle size敏感
2. SSR友好, `hydration`简单
3. 无需Provider包裹, 代码更干净
4. 中间件系统灵活(persist/immer/devtools)

### 6.2 Store架构

```
stores/
├── chat-store.ts      # 对话消息+流式状态
├── session-store.ts   # 会话列表+当前会话
├── auth-store.ts      # 用户认证状态
├── config-store.ts    # API配置+交易参数+策略+渠道
├── credits-store.ts   # 积分余额+明细
└── ui-store.ts        # 面板展开/折叠+主题
```

## 9. 数据库选型: PostgreSQL + Prisma

### 9.1 决策: PostgreSQL + Prisma ORM

**理由**:
1. 用户/积分/订单/策略配置等**结构化数据**, RDB天然适合
2. Prisma类型安全, 迁移管理, 批量用户场景可靠性高
3. 腾讯云CVM可直接部署, 或后期迁移到腾讯云PostgreSQL
4. 与Auth.js无缝集成

### 9.2 核心数据模型

```
User → CreditsAccount → Transaction (积分流水)
     → ApiConfig (API密钥, AES加密)
     → TradingParams (交易参数)
     → Strategy (策略配置)
     → ChannelConfig (通信渠道)
     → Session (对话会话)
```
